# Grid Style Engineering Practice

**Purpose:** 记录本轮 `Grid` 风格重构、调样、校验过程中沉淀下来的经验、方法与避坑原则。  
**Last updated:** 2026-04-15

## Why This Document Exists

`GRID-STYLE-REFERENCE.md` 负责回答“现在的 `Grid` 规则是什么”。  
本文负责回答另外三个问题：

1. 为什么要这样设计。
2. 这轮工作中哪些做法是有效的。
3. 以后继续改 `Grid` 时，怎样避免再次进入混乱状态。

## Core Decisions

### 1. Grid 和 Editorial 必须彻底隔离

这轮最大的教训，是两个风格如果继续共享结构假设、共享 DOM 语义、共享样式选择器，后续每一次微调都会互相误伤。

当前有效做法：

1. `Grid` 和 `Editorial` 只共享必要的抽屉内容挂载协议。
2. 两边各自拥有独立壳层、独立样式文件、独立 DOM 锚点。
3. `Grid` 相关改动，不允许再复用 `Editorial` 的标题、头部、侧栏语义节点。

工程结论：

1. 风格切换不只是“换皮肤”，而是“换壳层”。
2. 真正稳定的多风格系统，必须先拆边界，再做视觉变化。

### 2. 先固定结构，再调视觉

在 `Grid` 里，左侧和右侧结构必须先稳定，才能谈对齐和美术微调。

当前有效做法：

1. 左侧固定为 `toolbar / spacer / editor` 三段。
2. 右侧固定为 `header / spacer / content` 三段。
3. spacer 必须是显式容器，不允许继续拿 `margin-bottom` 假装结构留白。

工程结论：

1. 能用结构表达的间距，不要用“碰巧对上”的 margin 表达。
2. 显式 spacer 的维护成本，远低于散落 margin 的维护成本。

### 3. 几何关系必须变量化

这轮最明显的稳定点，是把 `Grid` 的顶部 inset、水平 inset、section gap、按钮大小统一进一组变量。

当前有效做法：

1. 左右两边共用同一组几何变量。
2. 按钮位置、标签栏位置、header/content 的外侧留白，全都从变量推导。
3. 任何微调优先改变量，不直接改散落数值。

工程结论：

1. 几何系统一旦出现“一个地方是 14，另一个地方还是 8”，后续就只能靠肉眼补洞。
2. 变量不是装饰，它是防止回归的边界。

## Singleton Layout Config

### 1. 统一配置源

当前 `Grid` 的间距和缩放，都通过单例配置源管理：

1. `editor-modules/grid-style/layout-config-store.js`
2. `editor-modules/grid-style/layout-manager.js`

职责分工：

1. `layout-config-store.js`
   持有单例配置、处理 `set / update / reset / subscribe`。
2. `layout-manager.js`
   纯粹负责把配置换算成 CSS 变量。

这套拆分非常重要，因为它保证：

1. 配置层只管“值是什么”。
2. 推导层只管“怎么换算”。
3. 页面层只管“怎么消费结果”。

### 2. 14px/16px 这类值的正确归属

这轮最终确认的原则是：

1. `14px`、`16px` 这种纵向基线，不应该散在 CSS 里。
2. 它应该是 `Grid` 单例配置里的 `shell_vertical_inset_base`。
3. `inner_margin / outer_margin / inner_outer_margin_consistent` 也属于同一层配置。

工程结论：

1. “默认值”也是配置，不是样式文件里的常量。
2. 只要后续还需要继续试视觉，默认值就必须可编程地被更新和回滚。

### 3. 自动更新，而不是手动同步

这轮有效的机制不是“改完配置再手动调用一次样式函数”，而是：

1. 页面订阅 `gridLayoutConfigStore`。
2. 配置更新后自动重算 CSS 变量。
3. 变量变化后自动触发布局重绘。

工程结论：

1. 只要还需要频繁试错，自动更新就不是可选项。
2. 如果更新链路需要手动补一步，迟早会忘。

## Practical Layout Lessons

### 1. Header 容器不要假装包住子元素

本轮出现过一次典型错误：

1. `header` 高度只有一行。
2. 子容器用 `absolute + top` 往下推。
3. 视觉上像是“按钮跑出了 header 盒子”。

后来确认，更稳的做法是：

1. 父容器自己承担布局留白。
2. 子容器回到正常流。
3. 自测去量真实几何，而不是看 DevTools 蓝框“像不像”。

### 2. Padding 和 Margin 的职责要分开

这轮有效的分工是：

1. 组件与外部容器的关系，用外层 inset 或显式 spacer 表达。
2. 内容区自己的壳内留白，用 content/editor 的 padding 表达。
3. 不允许把结构留白和内容留白混用。

工程结论：

1. 一旦 `padding-top` 和“假结构间距”混在一起，后面每次调对齐都很难判断是谁在起作用。

### 3. 小视觉调整也要走系统

后面做的这些微调虽然很小，但都遵循同一原则：

1. Toggle 两个按钮的缝隙缩小，只改按钮自身横向 margin。
2. `Styio` 标题放大，只改标题字号变量。
3. 不因为是“小改动”，就绕开现有几何和命名系统。

工程结论：

1. 小改动最容易让代码重新变脏。
2. 真正稳定的系统，要保证“小改动也只能走正路”。

## Testing Practice

### 1. 自测不能写死视觉常量

这轮把 `Grid` 自测从“写死 14px”改成了“读取当前 CSS 变量”。

这样做的好处：

1. 默认值从 `14` 改到 `16` 时，自测不用重写整个检查体系。
2. 运行时调 `outer_margin` 或 `inner_margin` 时，自测仍然有意义。

### 2. 自测要覆盖运行时更新链路

当前有效做法：

1. 先测静态几何关系。
2. 再通过 `window.__styioGridLayoutConfig.update()` 改配置。
3. 检查 CSS 变量是否变了。
4. 再量一遍实际几何。
5. 最后 `reset()`，确认能回到基线。

工程结论：

1. 只测初始化，不测更新链路，等于没有测完整。

### 3. 测试步骤要自己收尾

这轮自测里真实踩到过的坑：

1. 新增测试步骤后把侧边栏留在收起态。
2. 后续步骤仍按“设置区打开中”去点按钮。
3. 结果点击被主编辑区拦截。

最后确认的规则：

1. 每个复杂测试步骤结束时，要把页面收回到后续步骤可消费的状态。
2. 不要假设下一个步骤会自动修复前一个步骤留下的状态。

## Cache Busting Practice

这轮反复出现过“你已经改了，但页面没变”的情况。结论很直接：

1. 每次改 `editor.css / grid-style.css / editor.js` 这类静态资源，都要同步提版本。
2. 如果不提版本，就无法区分“代码没生效”和“视觉真的不对”。

工程结论：

1. 本地开发同样需要缓存策略。
2. 不处理缓存，调 UI 会浪费大量无效沟通。

## Recommended Workflow

后续继续改 `Grid`，建议严格按下面的顺序：

1. 先确认这次改的是结构、几何、还是纯视觉微调。
2. 如果是几何或间距，优先改单例配置源。
3. 如果是推导逻辑，再改 `layout-manager.js`。
4. 如果只是消费端排布，再改 `grid-style.css`。
5. 如果结构变化，再改 `editor.html`。
6. 补或改 `check-editor-load.mjs`。
7. 跑 `npm run selftest:editor`。
8. 若规则变化，更新 `GRID-STYLE-REFERENCE.md`。
9. 若方法论变化，更新本文。

## Prohibited Habits

后续不应该再做这些事：

1. 直接在多个选择器里同步手改同一组间距数字。
2. 为了对齐而偷偷加一个 `absolute + top` 偏移。
3. 让 `Grid` 和 `Editorial` 重新共享关键标题、header、drawer 语义节点。
4. 让测试依赖“当前正好是 14px/16px”这种写死常量。
5. 调完样式不提资源版本。
6. 只凭截图感觉判断对齐，而不看实际几何关系。

## Summary

这轮 `Grid` 工作沉淀下来的核心实践只有四条：

1. 先隔离风格边界，再做美术调整。
2. 先固定结构和几何系统，再调细节。
3. 配置集中到单例源，页面通过订阅自动更新。
4. 自测必须验证几何关系，也必须验证运行时更新链路。

只要这四条不破，后续继续改 `Grid`，成本会明显下降。
